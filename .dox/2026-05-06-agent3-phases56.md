## 4. Risk Assessment

### High Risk: Phase 0.5.1 Not Executed

If Phase 6 (SCIP ingestion) is implemented before Phase 0.5.1, every node with `file_type="code_index"` or `file_type="code_index_symbol"` will generate validation errors during `build_from_json()`. The runbook correctly flags this as a showstopper, but an implementer who skips Phase 0 will waste effort.

### Medium Risk: Line Number Drift

The runbook's line references for insertion points in `tests/test_extract.py` are slightly stale (~3 lines off for `test_cross_file_calls_skip_ambiguous_duplicate_labels`). Using the test function **names** rather than line numbers avoids this risk entirely. The runbook itself does both — it quotes line numbers for orientation but describes which function to append after, which is safer.

### Low Risk: `extract()` Cross-File Call Block Structure

The live code at `graphify/extract.py:4670-4719` implements the same conservative cross-file call resolution logic that the runbook proposes to extract into `symbol_resolution.py`. The behavior is equivalent:
- Skips rationale nodes (runbook Phase 3 says this; live code does this at line 4681)
- Skips member calls (line 4697)
- Skips ambiguous names (line 4703)
- Emits `INFERRED` with `confidence_score=0.8` (line 4714-4715)

The runbook's refactor would move this logic into a testable module without changing behavior. This is low-risk.

### No Risk: `build.py` Stability

All three build functions (`build_from_json`, `build`, `build_merge`) have signatures that match the runbook exactly. The internal flow (validate → build → dedup → return) is unchanged. No new validation checks have been added that would affect Phase 6 ingestion beyond the `VALID_FILE_TYPES` issue already documented.
